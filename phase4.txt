Task 7: Design Journal

The most difficult decision was choosing how to prioritize tasks during scheduling. Initially, I tried a simple greedy approach where tasks were assigned in arbitrary order. However, this led to many infeasible cases because high-conflict tasks were assigned too late.

I then switched to sorting tasks based on weight, conflict degree, and SLA tightness. The trade-off was between fairness and feasibility. Giving priority to high-weight tasks improved penalty but sometimes made scheduling harder.

I rejected a purely random or FIFO ordering because it ignored the structure of the problem.

In a Kafka system:

My algorithm can:

Understanding
One surprising thing I learned is that combining multiple constraints makes the problem much harder than solving them individually.

Graph coloring alone is manageable, and resource allocation alone is also manageable, but combining them with time windows creates a very complex system.

I also learned that greedy algorithms are not enough, and hybrid approaches are necessary.

I will explain my algorithm step by step

My algorithm works in a simple step-by-step way to assign tasks to slots without breaking any rules. First, I arrange all tasks in an order so that the most important and most difficult ones are handled first. 
Then, for each task, I try all possible slots that are allowed by its time window. For every slot, I check three things: whether it conflicts with any already assigned task, 
whether it fits within the resource limits of that slot, and whether it stays within its SLA window. If a slot satisfies all these conditions, I temporarily place the task there and calculate the penalty. 
After checking all possible slots, I choose the one with the lowest penalty and assign the task permanently. If no slot is possible, I try to adjust by moving some already assigned tasks to 
other slots. If even that does not work, I say that the assignment
is not possible. In this way, my algorithm tries to balance correctness and efficiency while keeping the penalty as low as possible.

What if slots have different capacities?
If different slots have different capacities, 
my algorithm already supports this. 
I store capacity separately for each slot, so while checking resource constraints, I compare the task’s requirement with that specific slot’s capacity. 
This means the algorithm naturally handles uneven resource distribution across slots without any modification.

One design decision I would change
If I had more time, I would improve the local repair mechanism. 
Currently, it only tries simple adjustments like moving one task at a time. 
In some difficult cases, this is not enough and the algorithm fails even when a valid solution exists. I would extend it to allow multiple task swaps or limited backtracking. This would increase the success rate, especially in dense conflict scenarios, although it would also increase computation time slightly.
